# Changelog

## [Unreleased]

### Changed

- Hid the Live TV and Music sections behind compile-time `FeatureFlags` (default off). The sidebar sections, settings rows, music home shelves, and music search results no longer appear. All code and SwiftData models stay in place, so on-disk stores remain valid and the features can be re-enabled by flipping a flag.
- The content-status label now includes "New Episode Every \<day\>" for currently-airing weekly shows — derived from the upcoming episode's air weekday when it falls within ~a fortnight of the previous one (a longer gap reads as "Returns \<date\>" instead). Specials (TMDb season 0) no longer misread as a new season.
- The content-status label now also appears on the TV show / movie **detail page** (above the title), not just the hero — resolved from the item's TMDb status once the detail loads. To make this work end to end, Plex requests now ask for external IDs (`includeGuids=1`) across hubs, recently-added, metadata, and children, so library/hero items reliably carry a TMDb id to look the status up by (previously the hero label could never resolve because home-hub items shipped without guids).
- The Home hero can now show an editorial content-status label above the title — "New Season \<month\>", "Returns \<date\>", "Premieres \<date\>", "Coming \<date\>", or "All Episodes Available" — derived from TMDb status data (`status`, `next_episode_to_air`, `seasons[]`, `release_date`) decoded from the detail payload the app already fetches. Dated labels show a specific day (e.g. "06 August") when a concrete air/release date is known. The label appears only when it is genuinely data-backed and still upcoming, and is hidden otherwise (no chip on missing or stale data, and no generic "Recently Added" noise on the hero). It reads as one VoiceOver element and stays stable while the hero auto-rotates. No Apple ranking/branding claims are shown. The data path is confirmed end to end (the TMDb-details proxy returns the status fields); a label still only shows when the item has a matched TMDb id and a genuinely upcoming/qualifying status.
- Home hero now auto-rotates through its eligible items on a deterministic ~10s interval, instead of only advancing when you press Next. Rotation continues while the hero is focused (only the slide content swaps; the Play / More Info button row stays put), matching a native hero — it pauses only while a play target is resolving, while a detail/preview/player/resume screen is presented, or while the app is backgrounded. Manual Next still works and resets the timer. Under Reduce Motion the slide changes instantly. No per-rotation network calls.
- Epic 3 PR3: the row preview now respects the system Reduce Motion setting — poster→preview, paging, expand, and collapse transitions apply instantly (no animation) without losing the destination, while full motion is unchanged otherwise. Preview exit and focus return-to-poster behavior is now covered by tests.
- Epic 2 PR5: Continue Watching is now pinned as the most prominent Home content row, and its cards expose a full VoiceOver description (title, episode, time remaining, progress).
- Epic 2 PR4: Home is now hero-first — a large cinematic hero leads the Home screen by default, favouring Continue Watching, then featured and recently added content, with Play / More Info actions. (Can still be turned off in settings.)

### Security

- Playback observability hardened and verified (E4-PR1). An exhaustive audit of every playback diagnostic (≈395 sites across the playback pipeline, player view model, and Live TV) found a single residual console leak — the FFmpeg demuxer logged the full stream URL (`url.absoluteString`, which carries `X-Plex-Token`) when opening the URLSession-backed AVIO path, and that debug log runs on the simulator. It now logs only the asset's last path component (no token, host, or query), matching the rest of the pipeline. Every other URL/token-touching log or Sentry sink was verified to carry only safe data (last path component, host, counts, or an explicit `[REDACTED_URL]`); no raw token or full stream URL reaches the console or Sentry. Regression tests (`PlaybackObservabilityTests`) lock these invariants. The original raw `stream_url` Sentry-extra finding (`E0-OBS-002`) is now closed; the broader observability finding (`E0-OBS-003`) is reduced to the non-security task of migrating debug `print()` to structured logging. See `Docs/modernization/epic-4/playback-observability-audit.md`.
- HLS manifest diagnostics no longer leak the Plex auth token. The HLS master/keyframe playlist debug logging (in `HLSManifestEnricher` and the player view model) previously printed the full manifest body, whose rewritten URLs carry `X-Plex-Token`; it now logs only a token-free structural summary (line/variant/track counts). Error and local-server-URL diagnostics on those paths are redacted as well. DEBUG-only logging, but simulator runs are DEBUG, so the token was reaching the console.
- Epic 2 PR7: Home error messages are now sanitized before display — a failed request's technical description (which can contain a token-bearing URL) can no longer appear in on-screen copy. Users see calm, plain-language errors; clean messages like "the connection appears to be offline" are preserved.
- Epic 2 PR2: Top Shelf no longer shares Plex token-bearing image URLs with the system extension. Artwork is handed off as local files in the App Group; the extension reads local files only. Top Shelf items and deep links are unchanged.

### Changed

- The Home hero and the show / movie detail page now carry a subtle adaptive background tint drawn from the dominant colour of the artwork already on screen — a quiet, Apple-TV-style sense of atmosphere rather than a flat backdrop (ADO-06). The colour is sampled once per item from the image the screen already loaded (no new network request, no new data source), off the main thread, and cached. The tint sits beneath the existing legibility scrims and is always gentle, so text contrast is never reduced below the current baseline; it is muted so a bright poster can't wash out the screen. Accessibility-aware: it is dropped entirely under Increase Contrast (falls back to the current dark backdrop), reduced in intensity under Reduce Transparency, and appears without an animated fade under Reduce Motion.
- Content ratings (e.g. PG-13, TV-MA, MA15+) now render as a first-class metadata badge — a small rounded chip consistent with the technical-format badges (4K / Dolby Vision / Atmos) — on the hero and detail metadata rows, instead of an ad-hoc outlined label (ADO-06). The rating value and its source are unchanged; only the display is normalised (a leading locale prefix like "US:" is stripped so it reads "TV-MA", not "US:TV-MA"). Landscape shelf cards already include the rating in their info line, so no duplicate badge is added there. The metadata hierarchy (status → title → metadata → synopsis → actions) is unchanged.
- Episode cards (TV show / season detail) now show a restrained, Plex-backed content-status chip above the title — "Season Finale", "New Episode Today", or "New Episode" — when the episode's own metadata supports it (ADO-05). This completes the Content Status Label System across its last placement (episode cards), alongside the hero and detail page. The chip is truthful by construction: Season Finale appears only when the episode is the final numbered episode of a regular season (specials / season 0 are excluded, and missing season/episode/count data shows nothing); "New Episode Today" / "New Episode" come only from a present, non-future original air date (`originallyAvailableAt`). No show-level labels (New Season / Returns / Premieres / Coming Soon / All Episodes Available / New Episode Every \<day\>) ever appear on a per-episode card — those stay on the hero/detail. No new network calls, no provider/playback changes: the season episode count is computed from the already-loaded episode list and the air date rides on the existing metadata. The chip reads as part of the episode's single VoiceOver summary only when visible (number → title → status → runtime → state), and nothing is announced when no chip is shown.

- Playback fallback policy corrected and wired to match the real HLS-fallback behaviour (E4-PR5C): wiring the policies in E4-PR5B exposed that `PlaybackFallbackPolicy` had its player branches inverted — it claimed the AVPlayer path had no automatic fallback and the RPlayer path fell back to HLS, when the live code is the opposite. An audit of every fallback path confirmed the truth: the AVPlayer path reloads as Plex HLS exactly once on a direct/remux startup or item failure (gated by HLS availability + a one-shot guard), while the RPlayer path surfaces a calm error with no automatic route fallback (the user can retry). The policy was corrected to that model and then wired as the single decision source for the AVPlayer fallback (`shouldAttemptRivuletFallbackOnItemFailure` and both startup-failure catches now consult it; the concurrency/dedup mechanics stay at the call sites), so the behaviour is identical to the previous `planHasHLSFallback && !alreadyAttempted` gate — proven by integration tests across every family × HLS-availability × attempted combination. A safe `routeFellBack` telemetry event (anonymised from/to route names + typed failure category, no URL/token) is now emitted when the one-shot fallback fires. No route-default change, no AVKit-first flip, no playback/subtitle/audio/chapter/post-play/UX change. `DEBT-E4-PR3-FALLBACK-001` is closed; `DEBT-E4-PR3-001` is reduced (only the `ContentRouter` route-family replacement remains deferred, as its reasoning strings + URL construction are pinned by tests). See `Docs/modernization/epic-4/playback-routing-policy.md`.
- Playback policy seams wired into live code without an AVKit default flip (E4-PR5B): the pure Epic 4 playback policies are no longer built-but-unused — four decision seams now consult them as the single source while runtime behaviour stays identical. Behaviour-preservation is locked by new `PlaybackPolicyIntegrationTests`. Deferred with documented reasons: the `ContentRouter` route-family decision (its reasoning strings + URL construction are pinned by tests), the RPlayer async route-change/AirPlay-instability recovery ladder (timing-sensitive; needs on-device AirPlay validation), and the start-offset seek arithmetic (kept in seconds to avoid sub-millisecond drift). Integration also surfaced a model bug: `PlaybackFallbackPolicy` treats the AVPlayer path as having no automatic fallback, but the live AVPlayer path does fall back to HLS once — so the policy was deliberately *not* wired into the fallback path (that would change behaviour); the correction is tracked as `DEBT-E4-PR3-FALLBACK-001`. The accumulated integration debt (`DEBT-E4-PR2/3/4/5-001`) is reduced, not closed; the AVKit-first default flip remains deferred to E4-PR6, gated on the media-validation corpus + a physical Apple TV. No routing, player-default, subtitle, audio, chapter, post-play, UX, or project-setting change. See `Docs/modernization/epic-4/playback-routing-policy.md`, `playback-resume-policy.md`, `playback-interruption-recovery-policy.md`, and `playback-telemetry-contract.md`.
